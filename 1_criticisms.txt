I read the v6 design note and the implementation notes first, then unpacked, built, and ran the prototype. Overall verdict: this is a serious resident-layer prototype with a sound architectural core, but it is not yet fully compliant with its strongest v6 claims. In particular, the broad “zero allocation / zero code path / zero overhead” story is only partially true in the current code, and layout B coarse has at least two real correctness problems. The design note also explicitly keeps the layout decision open and says the transient AoSoA tile layer, GPU kernels, tree walk, and MPI wiring are still Phase 2+, so this code should be judged as a resident-storage prototype, not as proof of final end-to-end HPC performance.   

What is good, and largely faithful to the design, is the overall structure. The split into `opg_common/` plus three independent layouts is present. `PhysicsConfig` as a C++20 NTTP, concepts, `requires`-constrained accessors, `optional_ptr` / `optional_field`, and the stateless-lambda `ArrayRegistry` are all implemented in the style the notes describe. The pack/unpack path in coarse B also correctly uses `std::memcpy`, matching the implementation note’s discussion of padding-sensitive byte transport. I also verified that the baseline build works in this environment and the example programs run.   

The first major problem is the reshuffle scratch path. In `opg_common/include/opg_common/permutation/permutation.hpp:97-103`, `apply_permutation` takes `T* temp`, and `register_array` later casts the untyped scratch pointer back to `T*` at `permutation.hpp:121-124`. But the examples pass scratch as raw `malloc` storage in `examples/example_layout_b.cpp:93-97` and as `std::vector<std::byte>` storage in `examples/example_layout_b_coarse.cpp:214-224`. That is not guaranteed to be aligned for `alignas(32)` or `alignas(64)` particle types such as `GasCore` and `PCoreB` (`opg_layout_b_coarse/include/opg_layout_b_coarse/particles.hpp:51`, `:118`). So this is undefined behavior. There is also a second, subtler issue here: `apply_permutation` gathers with assignment, not `memcpy`, so it does not inherit the padding guarantee that you correctly enforced in coarse-B pack/unpack. The implementation notes explicitly call out that aggregate assignment is not obliged to preserve trailing padding for bitwise-reproducible transport; the same caveat applies here if you ever care about deterministic byte images after reshuffle. 
The fix is to change the reshuffle primitive to operate on byte scratch, not `T*` scratch: gather each element with `std::memcpy(tmp_bytes + i*sizeof(T), array + perm[i], sizeof(T))`, then `memcpy` the whole block back. That simultaneously removes the alignment trap and makes padding behavior explicit.

The second major problem is a determinism regression in coarse B. The design note and implementation notes both say that v6 fixed sort determinism by tiebreaking on `original_idx`, and they even describe this as the authoritative rule for restart reproducibility. But `opg_layout_b_coarse/include/opg_layout_b_coarse/reshuffle.hpp:78-82` sorts by `(key, type)`, not `(key, original_idx)`. That means equal-key equal-type elements are still free to reorder arbitrarily, so coarse B does not actually implement the determinism rule described in the documents. This also duplicates logic that already exists correctly in `PHKeySorter::sort_by_key()` in `opg_common/include/opg_common/permutation/permutation.hpp:232-238`.  
The fix is simple: coarse B should call the shared `PHKeySorter::sort_by_key()` or copy that exact comparator. If you want type grouping, do it as a separate stable in-leaf sub-sort, not as a replacement for the primary deterministic comparator.

The third major problem is a real coarse-B mutation bug. The design note says coarse B mutation is implemented and smoke-tested, with type sub-sort preserved after gas→star mutation. But `opg_layout_b_coarse/include/opg_layout_b_coarse/mutation.hpp:163-185` only decrements `type_count[Gas]`, walks through intermediate DM types, and then adjusts `type_offset[Star]` and `type_count[Star]`. It does not recompute the offsets of DM and BH runs. I reproduced this with a toy leaf containing gas, DM, and star particles: after mutation the common-array particle order was corrected, but the leaf descriptor’s `type_offset[]` bookkeeping was stale, so the DM sub-range could include a star. That is a correctness bug, not just a TODO. It directly contradicts the v6 claim that coarse-B mutation preserves the leaf type sub-sort.  
The fix is not to hand-edit one offset. After any in-leaf mutation, recompute all `type_offset[]` from the canonical per-type counts, in canonical order. A helper like `recompute_type_offsets(BoxLeafBase&)` is safer than local arithmetic.

The fourth problem is that compile-time feature gating is not as complete as the documents imply. The design requirement is “disabled modules → zero allocation, zero code path.” The implementation notes likewise say truly zero-overhead disabled features require five layers: config, concept, accessor, storage, allocation. In code, that is true for some features, but not all. In both fine layouts, gas core storage is always allocated: `opg_layout_b/include/opg_layout_b/particle_container.hpp:150-173` and `opg_layout_c/include/opg_layout_c/particle_container.hpp:150-157`. In coarse B, `gas_all_` is always allocated at `opg_layout_b_coarse/include/opg_layout_b_coarse/particle_container.hpp:152`. Meanwhile the `HasHydro` concept exists in `physics_config.hpp:243-245` but is never used anywhere in the codebase. So GravityOnly does not actually mean “no hydro allocation”; it only disables some optional hydro-related substructures.  
That may be an acceptable design choice, but then the docs should say “zero overhead for optional subfeatures inside the chosen resident layout,” not “zero allocation” globally. Or, if the stronger statement is intended, gas base storage should be gated on `HasHydro<Cfg>`.

There is a related gating bug in `PhysicsConfig::has_metals()`. In `opg_common/include/opg_common/types/physics_config.hpp:150-152`, it returns
`stellar_evolution || (star_formation && n_metal_species > 0)`.
That means any config with `stellar_evolution=true` is treated as “has metals” even when `n_metal_species==0`. I verified that a custom config with `stellar_evolution=true` and `n_metal_species=0` still satisfies `HasMetals`, so the code allocates `GasMetal<0>` / `StarCore<0>` storage paths instead of disabling them. That is not consistent with the design-note story that `OPG_NMET` is the single source of truth and that feature presence should be compile-time-gated from actual config values.  
The fix should be something like `return n_metal_species > 0 && (stellar_evolution || star_formation);`, or better, split the notion into separate gas-metal and star-metal concepts so the semantics are explicit.

A fifth weakness is that the advertised config surface is ahead of the actual storage model. `HasCosmicRays` exists in `physics_config.hpp:276-277`, and both fine-layout containers define `GasCRT` aliases at `opg_layout_b/.../particle_container.hpp:51` and `opg_layout_c/.../particle_container.hpp:50`, but no layout allocates a CR array, exposes a CR accessor, or includes CR in mutation/reshuffle paths. Similarly, the fine layouts allocate `GasMFM`, but coarse B’s `GasAllB<Cfg>` at `opg_layout_b_coarse/include/opg_layout_b_coarse/particles.hpp:181-188` has `core`, `grad`, `mag`, `metals`, and `sfr` only: there is no MFM block at all. So `MFMHydro` is not implemented consistently across the three layout candidates, which weakens the promise that the benchmark apparatus compares identical physics under identical `PhysicsConfig`s.  
The fix should be binary: either wire MFM/CR fully into every layout, or make unsupported configs fail at compile time in that layout so there is no silent under-modeling.

A sixth weakness is extensibility around mutation. The design note makes a big point that registration localizes the reshuffle contract. But `opg_layout_c/include/opg_layout_c/mutation.hpp:151-175` does not use the gas registry to swap “all registered gas arrays”; it hand-swaps a hard-coded list of known gas arrays. That works today, but it is exactly the sort of place where adding a future `GasCR` or other gas block will silently break mutation unless someone remembers to update this function. That is contrary to the design’s extensibility goal. 
The fix is to add a `swap_all(a,b)` operation to `ArrayRegistry`, parallel to `reshuffle_all`, so mutation can say “swap all gas arrays” once rather than re-enumerating the list.

A seventh issue is API gating on mutation. Both `opg_layout_c/include/opg_layout_c/mutation.hpp:97-101` and `opg_layout_b_coarse/include/opg_layout_b_coarse/mutation.hpp:95-99` require only `HasStellarEvolution<Cfg>`. That means gas→star mutation APIs are available even when star formation is disabled but stellar evolution is enabled. Mechanically, star storage exists, but semantically this is the wrong compile-time contract. New star creation should require `HasStarFormation<Cfg>` as well, otherwise the type system is not actually preventing use of a disabled physics path.

A few smaller cleanup points are worth mentioning. `ArrayRegistry::register_array()` silently drops arrays when `count_ >= MaxArrays` (`permutation.hpp:115-126`), and `reshuffle_all()` silently skips arrays if `e.elems < n` (`permutation.hpp:141-146`). For a codebase where “forgetting one array” is a canonical corruption mode, silent failure is the wrong policy. Make these debug assertions at minimum, or hard errors. Several public headers do `using namespace opg::common;` inside their namespace blocks; that is tolerable in a prototype but poor header hygiene for a real library. `total_bytes_allocated()` in the fine-layout containers sums `N*sizeof(T)` but does not include arena alignment padding, so it under-reports actual footprint. And while the advertised warning set is clean, stricter warnings (`-Wconversion`, `-Wsign-conversion`, `-Wshadow`, etc.) surface a fair number of sign/size conversion issues that are worth cleaning before you trust large-scale HPC builds.

On your specific question “is it optimal in terms of C++ usage?”, my answer is: mostly good C++20 choices, with some overstatement. The NTTP/concepts/`requires` machinery is exactly the right tool for compile-time physics policies. The structs are POD-like, the arena is simple, there is no vtable-heavy design, and the code is readable. The weak spots are mostly around edge-case rigor and internal consistency, not around using “fancy C++ for its own sake.” The only place where I would push back on the prose is the claim that GCC “inlines across the indirect call” in the registry loop. Maybe, sometimes, on a given build. But the architectural truth is what the code says: one function-pointer call per registered array per reshuffle. That is low overhead, but it is still runtime overhead. 

On “is it zero runtime overhead?”, the honest answer is no, not globally. It is zero-overhead in some important senses: disabled accessors really disappear, many disabled optional storages really collapse to empty types, and allocation is compile-time-gated for a good subset of features. But the registry reshuffle has a runtime indirect call by design, and the coarse layout’s fixed `alignas(64)` common structs mean disabled optional subfields do not necessarily reduce bandwidth. In a quick size check, `sizeof(PDynB<GravityOnly>)`, `sizeof(PDynB<StandardSPH>)`, and `sizeof(PDynB<FullMHD>)` were all 64 bytes; same for `PAuxB`. So `[[no_unique_address]]` helps the interface and layout of subobjects, but not all runtime memory traffic. The current code is therefore “low-overhead and type-safe,” not literally “zero-overhead” in the strongest possible reading.  

On “will it compile to HPC code?”, as a resident-layer prototype, yes: it compiles cleanly with the advertised flags, and the generated code structure is the kind of thing an HPC compiler understands well—plain arrays, POD-ish structs, explicit memory layout, no hidden ownership graph. But this prototype does not yet establish the actual HPC path that matters most: transient tiles, GPU kernels, shared interaction lists, GPU sort, and MPI layer are explicitly out of scope in v2. So the right conclusion is: the resident data model looks plausible for HPC, but the performance-critical proof still lives in the missing Phase 2 work.  

The order I would fix things is straightforward. First, rewrite `apply_permutation` to use byte-wise `memcpy` scratch and make scratch alignment explicit or irrelevant. Second, remove the coarse-B sort duplication and route it through the shared deterministic sorter. Third, repair coarse-B mutation bookkeeping by recomputing all leaf offsets, then add a regression test that checks both particle order and `type_offset/type_count` invariants. Fourth, fix the gating semantics: `HasMetals`, hydro allocation policy, and mutation `requires` clauses. Fifth, either implement or hard-disable MFM/CR on layouts that do not truly support them. Sixth, turn the registry’s silent skips into assertions/errors. Only after those are done would I trust benchmark results enough to compare B vs B′ vs C.
